## 1. Objetivo

Chunking genérico funcional por tokens que produce chunks utilizables para
embedding/RAG. Primera pieza del pipeline que escribe en `document_chunks`.
Diseñado para español jurídico chileno.

**Principio rector:** Nunca dejar un documento sin chunks. Nunca cortar
contenido legal en un punto que cause pérdida de contexto para la IA.

---

## 2. Refinamientos sobre el Kanban original

### 2.1 Protected prefixes: fronteras legales básicas

Sin esperar el section-detector completo (7.07c), el chunker respeta
fronteras legales mínimas que, si se cortan, causan alucinación directa:

- `VISTOS:` / `CONSIDERANDO:` / `SE RESUELVE:` (estructura sentencia Art.170 CPC)
- `PRIMERO:` / `SEGUNDO:` / `TERCERO:` ... `DÉCIMO` (numeración considerandos)
- `EN LO PRINCIPAL:` / `PRIMER OTROSÍ:` / `SEGUNDO OTROSÍ:` (estructura escritos)
- `POR TANTO,` / `A SS. PIDO:` / `SOLICITO A SS.:` (petitorios)
- `CERTIFICO:` (actuaciones receptor)

El chunker usa estos como **puntos de corte prioritarios**. Si un
protected prefix cae dentro del rango de tokens, corta ahí.

### 2.2 Overlap semántico

En vez de retroceder exactamente 15% de tokens, el overlap retrocede
hasta el boundary de oración más cercano (punto seguido + espacio, o
doble newline). Si ese boundary cae dentro del rango 10-20% del target,
usarlo. Si no, fallback al mecánico.

### 2.3 Aproximación de tokens para español jurídico

`chars/4` sobreestima tokens en español (~20%). Se usa `chars/5` como
base y se cuenta por palabras como validación cruzada:
- `estimateTokens(text) = Math.ceil(text.length / 5)`
- Validación: `words.length * 1.3` debería dar resultado similar

### 2.4 Metadata del normalizer propagada a cada chunk

Cada chunk hereda del normalizer:
- `page_number`: calculado mapeando el offset del chunk contra `pageBoundaries[]`
- `foja`, `juzgado`, `rol`, `caratulado`: copiados del normalizer metadata
- `nomenclaturas`: del header del documento

### 2.5 Documentos cortos: no fragmentar

Documentos con < 200 tokens → 1 solo chunk, sin overlap, sin fragmentar.
Aplica a: certificados de firma, actuaciones de receptor breves, autos.

## 3. Parámetros de configuración

```
TARGET_TOKENS = 768          // optimal para text-embedding-004
TOKEN_FLEX = 0.30            // ±30% → rango real: 538-998
OVERLAP_PERCENT = 0.15       // 15% → ~115 tokens
MIN_CHUNK_TOKENS = 50        // merge con adyacente si menor
MAX_CHUNK_TOKENS = 1500      // subdividir si mayor
SHORT_DOC_THRESHOLD = 200    // docs más cortos → 1 chunk
CHARS_PER_TOKEN_ES = 5       // aprox para español jurídico
```

---

## 4. Schema destino (document_chunks)

```sql
chunk_index    int       -- 0-based, orden dentro del documento
chunk_text     text      -- texto puro del chunk (para display y LLM)
page_number    int       -- página del documento (de pageBoundaries)
section_type   text      -- 'general' (default, hasta que 7.07c lo refine)
metadata       jsonb     -- { foja, rol, juzgado, caratulado, overlap_prev, ... }
```
