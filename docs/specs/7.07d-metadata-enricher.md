## 1. Objetivo

Enriquece cada chunk con metadata completa heredada del documento padre,
la causa y el section-detector. Además, provee `buildEmbeddingInput()`
que genera el string enriquecido para text-embedding-004.

**Crítico para:** sistema de citas (3.09), RAG accuracy, anti-alucinación.

---

## 2. Tres capas de metadata

### Capa 1: Documento padre (processing_queue.metadata)
- `document_type`: sentencia/escrito/actuación/resolución/folio/directo/anexo
- `folio_numero`: número del folio en el expediente
- `cuaderno`: Principal, Excepciones Dilatorias, etc.
- `fecha_tramite`: fecha del trámite
- `desc_tramite`: "Sentencia", "Mero trámite", "Notificación", etc.
- `foja`: número de foja

### Capa 2: Detección (normalizer + section-detector)
- `section_type`: vistos/considerando_n/resolutivo/petitorio/otrosi/etc.
- `section_label`: "Considerando 6", "Petitorio"
- `section_confidence`: 0.0-1.0
- `page_number`: de pageBoundaries

### Capa 3: Causa (cases table)
- `procedimiento`: ordinario/ejecutivo/sumario/monitorio/voluntario
- `libro_tipo`: c/v/e/a/f/i
- `tribunal`: nombre del tribunal (complementa lo del normalizer)

---

## 3. buildEmbeddingInput()

Genera string: `[prefijo contextual] + chunk_text`

Prefijo: `[tipo_doc | sección | procedimiento | folio N | cuaderno X]`
Máximo ~25 tokens para no diluir el vector.

Solo incluye sección si confidence >= 0.70 (anti-alucinación).
chunk_text permanece puro en DB (para display al LLM y usuario).
