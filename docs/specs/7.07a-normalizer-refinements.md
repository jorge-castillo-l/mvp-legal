> **Fecha:** 2026-03-18

> **Módulo destino:** `src/lib/pipeline/chunking/normalizer.ts`
> **Dependencia:** 7.05 (Orchestrator) — completado
> **Próximo consumidor:** 7.07b (Token-Based Chunker)

---

## 1. Objetivo

Módulo que limpia artefactos digitales del sistema judicial PJUD/OJV del texto
extraído de PDFs, **antes** del chunking. Opera sobre `extracted_texts.full_text`
(el texto original se preserva intacto en DB como referencia).

**Principio rector:** Máxima fidelidad al texto legal original. Todo lo que se
remueva debe ser un artefacto digital del OJV/PJUD, nunca contenido jurídico.
Un falso positivo (remover texto legal confundiéndolo con artefacto) es mucho
peor que un falso negativo (dejar un artefacto en el texto).

---

## 2. Refinamientos sobre el Kanban original

Estos 5 refinamientos se aplican sobre la descripción de 7.07a en
`Kanban_MVP_V1.csv` para maximizar robustez y prevenir alucinaciones en la IA
downstream.

### 2.1 Paradigma "Extraer + Etiquetar + Reducir" (no solo "Remover")

El Kanban dice "detectar → extraer metadata útil → remover del texto". Se
refina: en vez de borrar completamente ciertos artefactos, **reemplazarlos
con etiquetas compactas** cuando el artefacto contiene información
jurídicamente relevante.

**Ejemplo — Firma Electrónica:**

```
ANTES (40-80 líneas de ruido criptográfico):
  Firmado electrónicamente por: CARLOS ANDRÉS PÉREZ ROJAS
  Juez Titular
  Séptimo Juzgado Civil de Santiago
  Fecha: 15/03/2026 14:30:22
  Código de verificación: a7b3c9d2e1f0...
  Ley N° 20.886 sobre Tramitación Electrónica
  Oficina Judicial Virtual del Poder Judicial

DESPUÉS (metadata extraída + texto limpio):
  Metadata JSONB: {
    firmante: "Carlos Andrés Pérez Rojas",
    cargo: "Juez Titular",
    fecha_firma: "2026-03-15T14:30:22"
  }
  Texto: (bloque completo removido)
```

**Ejemplo — Certificado de Ingreso:**

```
ANTES:
  CERTIFICADO DE INGRESO
  Corte Suprema
  ROL: C-1234-2025
  Fecha: 10/01/2025  Hora: 14:35

DESPUÉS (metadata extraída):
  Metadata JSONB: {
    certificado_ingreso: {
      rol: "C-1234-2025",
      fecha: "2025-01-10",
      hora: "14:35"
    }
  }
  Texto: (bloque completo removido)
```

**Razón:** Si removemos la firma sin capturar el firmante, la IA no puede
responder "¿Quién dictó la sentencia?" sin alucinar. Si removemos el
certificado sin capturar la fecha/hora de ingreso, perdemos datos para
cómputo de plazos.

### 2.2 Awareness del método de extracción

La función recibe `extraction_method: 'pdf-parse' | 'document-ai'` como
parámetro. Los pasos de limpieza se ajustan:

| Paso | pdf-parse | document-ai |
|------|-----------|-------------|
| 1. Headers repetidos | ✅ | ✅ |
| 2. Firmas electrónicas | ✅ | ✅ |
| 3. Certificado de Ingreso | ✅ | ✅ |
| 4. Foliatura suelta | ✅ | ✅ |
| 5. Normalizar encoding | ❌ (no necesario) | ✅ (tildes/ñ rotas) |
| 6. Colapsar whitespace | ✅ (suave) | ✅ (agresivo) |
| 7. Ghost text OCR | ❌ | ✅ |

**Razón:** Aplicar corrección de encoding a texto nativo de pdf-parse puede
corromper contenido legítimo. Document AI introduce artefactos específicos
del OCR que pdf-parse no tiene.

### 2.3 Conservadurismo como política por defecto

Cada patrón regex tiene dos variantes:

- **Estricto** (alta confianza): se aplica automáticamente.
- **Amplio** (baja confianza): solo se aplica si `mode: 'aggressive'`.
  En modo default, los matches amplios se **reportan pero no se remueven**.

Regla: **ante la duda, preservar el texto.** Un header de tribunal que queda
en el embedding genera ruido menor. Un considerando que desaparece genera
alucinación grave.

### 2.4 Tracking de límites de página

Antes de remover headers por página, el normalizer:

1. Detecta los boundaries de página (headers repetidos = indicadores de
   cambio de página).
2. Almacena los offsets como metadata:
   `{ page_boundaries: [0, 1543, 3201, ...] }`
3. Luego remueve los headers.

**Razón:** El sistema de citas (3.09) necesita saber "este chunk viene de
la página 5" para generar citas como *"(pág. 5)"*. Sin page boundaries,
esa información se pierde al colapsar el texto.

### 2.5 Modo auditoría (dry-run)

La función acepta `{ dryRun: true }` y en ese caso:

- NO modifica el texto.
- Retorna el texto original **anotado** con markers de lo que se removería.
- Incluye lista detallada de artefactos detectados con offsets.

```typescript
// Ejemplo de output en dry-run:
{
  annotatedText: "««HEADER»» PODER JUDICIAL... ««/HEADER»» Santiago, ...",
  detectedArtifacts: [
    { type: 'header', offset: 0, length: 89, confidence: 0.98 },
    { type: 'firma_electronica', offset: 4521, length: 342, confidence: 0.95 },
  ]
}
```

**Razón:** Permite validar los regex con fixtures reales antes de deployar,
e iteración rápida. El abogado puede revisar si se está comiendo texto legal.

## 3. Interfaz del módulo

```typescript
// src/lib/pipeline/chunking/normalizer.ts

type ExtractionMethod = 'pdf-parse' | 'document-ai'

interface NormalizerOptions {
  extractionMethod: ExtractionMethod
  dryRun?: boolean        // default: false
  mode?: 'conservative' | 'aggressive'  // default: 'conservative'
}

interface ExtractedArtifact {
  type: 'header' | 'firma_electronica' | 'certificado_ingreso' | 'foliatura' | 'encoding_fix' | 'ghost_text'
  offset: number
  length: number
  confidence: number
  rawText: string
  extractedData?: Record<string, unknown>
}

interface NormalizerResult {
  cleanText: string
  extractedMetadata: {
    tribunal?: string
    firmantes?: Array<{
      nombre: string
      cargo?: string
      fechaFirma?: string
    }>
    certificadoIngreso?: {
      rol?: string
      fecha?: string
      hora?: string
    }
    pageBoundaries: number[]
  }
  artifacts: ExtractedArtifact[]
  stats: {
    originalLength: number
    cleanLength: number
    reductionPercent: number
    artifactsDetected: number
    artifactsRemoved: number   // = detected si mode=aggressive; < detected si conservative
    encodingFixes: number       // solo si document-ai
  }
}

// Dry-run output
interface NormalizerDryRunResult {
  annotatedText: string
  detectedArtifacts: ExtractedArtifact[]
  stats: NormalizerResult['stats']
}

export function normalizePjudText(
  rawText: string,
  options: NormalizerOptions
): NormalizerResult | NormalizerDryRunResult
```

## 4. Pipeline de limpieza (7 pasos)

Cada paso es una función pura e independiente, ejecutadas en secuencia.
El output de un paso es el input del siguiente.

### Paso 1: `detectAndRemoveHeaders()`
- Detecta bloques repetidos: "PODER JUDICIAL" + "REPÚBLICA DE CHILE" + nombre tribunal.
- Extrae nombre del tribunal como metadata (antes de eliminar).
- Registra page boundaries en los offsets donde aparecen headers.
- Regex estricto: match del bloque completo con las 3 líneas en secuencia.

### Paso 2: `extractAndRemoveSignatures()`
- Detecta bloques de firma electrónica avanzada (Ley 20.886).
- Patrones: "Firmado electrónicamente por..." + timbres + códigos alfanuméricos OJV.
- Extrae: nombre firmante + cargo + fecha/hora de firma.
- Elimina: todo el bloque criptográfico (códigos de verificación, disclaimers Ley 20.886).

### Paso 3: `extractAndRemoveCertificadoIngreso()`
- Detecta el recuadro de Certificado de Ingreso de la OJV.
- Patrones: "Certificado de Ingreso" + "Corte Suprema" + ROL + Fecha + Hora.
- Extrae ROL + fecha + hora como metadata.
- Elimina el bloque completo.

### Paso 4: `removeFoliatura()`
- Elimina números de folio sueltos en esquinas de página.
- Patrones: números aislados (1-4 dígitos) al inicio/final de línea sin contexto.
- Conservador: solo remueve si el patrón es inequívoco (número solo en su línea).

### Paso 5: `normalizeEncoding()` ← solo document-ai
- Repara secuencias de encoding roto por OCR de Document AI.
- Map de correcciones conocidas: Ã¡→á, Ã©→é, Ã³→ó, Ãº→ú, Ã±→ñ, etc.
- Detecta primero si el texto tiene indicadores de encoding roto antes de aplicar.
- NO se aplica a output de pdf-parse (riesgo de corrupción).

### Paso 6: `collapseWhitespace()`
- Colapsa whitespace excesivo y saltos de línea irregulares.
- Preserva separadores de párrafo semántico (doble salto de línea).
- Agresividad varía: pdf-parse (suave) vs document-ai (agresivo, más basura).

### Paso 7: `removeOcrGhostText()` ← solo document-ai
- Elimina texto fantasma residual de OCR (marcas de agua leídas como texto).
- Patrones: strings muy cortos repetidos + baja densidad de caracteres legibles.
- Solo se aplica a document-ai. Muy conservador para evitar falsos positivos.

## 5. Punto de integración

Se integra en `src/lib/pipeline/orchestrator.ts`, después del paso 6
(guardar en extracted_texts) y antes del chunking (7.07b):

```
Paso 6: Guardar en extracted_texts (full_text ORIGINAL, intacto)
    ↓
Paso 7 (NUEVO): normalizePjudText(fullText, { extractionMethod })
    ↓
Paso 8 (7.07b): tokenChunker(normalizedText) → INSERT document_chunks
```

`extracted_texts.full_text` NUNCA se modifica. El texto limpio pasa
directamente al chunker. La metadata extraída de artefactos se almacena
en `document_chunks.metadata` JSONB.

## 8. Catálogo de artefactos reales (Capa 1 — Análisis de 15 fixtures)

> Análisis realizado el 2026-03-18 sobre fixtures en `docs/fixtures/normalizer/`.
> Fixtures: 15 documentos de 2 causas (C-15200-2023 17ºJCS + C-153-2021 Illapel),
> mezcla de pdf-parse y document-ai, cubriendo sentencias, escritos, actuaciones
> receptor, certificados, informes OCR, resoluciones.

### 8.1 ARTEFACTO: Firma electrónica PJUD (verificadoc.pjud.cl)

**Encontrado en:** fixtures 01, 06, 07, 10, 11, 15
**Método:** pdf-parse y document-ai
**Frecuencia:** En casi todo documento judicial. Aparece al final de cada página.

**Patrones reales:**

```
Código: XXDXXMLXXZH Este documento tiene firma electrónica y su original puede ser validado en http://verificadoc.pjud.cl
```

```
Cvc XXJZVDDYEX Este documento tiene firma electrónica y su original puede ser validado en http://verificadoc.pjud.cl o en la tramitación de la causa.
```

**Variantes del código:**
- Prefijo `Código:` (más común)
- Prefijo `Cvc` (variante Illapel)
- Prefijo `SEC Código:` (visto en previews)
- Códigos: 11 caracteres alfanuméricos mayúsculas (`XXDXXMLXXZH`, `VXZGXKXPJDX`,
  `NMJRXMBRFJT`, `DCYJXXXTXFZ`, `YEGEBRHKJXN`, `XXJZVDDYEX`)
- Sufijo opcional: `o en la tramitación de la causa.`

**Acción:** EXTRAER código como metadata → REMOVER bloque completo.
**Metadata:** `{ firma_pjud: { codigo: "XXDXXMLXXZH", url: "verificadoc.pjud.cl" } }`

### 8.2 ARTEFACTO: Marcador de página ("-- N of M --")

**Encontrado en:** todos los fixtures excepto 14 (que usa `-N-`)
**Frecuencia:** Al final de cada página de cada documento.

**Patrones reales:**

```
-- 1 of 9 --
-- 2 of 9 --
-- 1 of 1 --
-- 15 of 15 --
```

Variante OCR (fixture 14): `-1-`, `-2-`, `-3-`

**Acción:** EXTRAER total_pages del último marcador → REMOVER todos.
**Metadata:** `{ total_pages: 9 }` (del `M` en `N of M`)
**Uso crucial:** Estos marcadores indican boundaries de página. Registrar offsets
ANTES de removerlos para `page_boundaries[]`.

### 8.3 ARTEFACTO: Header NOMENCLATURA + JUZGADO + ROL + CARATULADO

**Encontrado en:** fixtures 01, 06, 10, 11, 15
**Método:** pdf-parse (documentos generados por el sistema PJUD)

**Patrón con FOJA:**
```
FOJA: 31 .- .- NOMENCLATURA : 1. [40]Sentencia JUZGADO : 17 Juzgado Civil de Santiago º CAUSA ROL : C-15200-2023 CARATULADO : MINERIA Y CONSTRUCCIONES CERRO ALTO LTDA./MADERAS ARAUCO S.A
```

**Patrón sin FOJA:**
```
NOMENCLATURA : 1. [4764]Aud. De Contes. Concil. y Prueba JUZGADO : 17 Juzgado Civil de Santiago º CAUSA ROL : C-15200-2023 CARATULADO : MINERIA Y CONSTRUCCIONES CERRO ALTO LTDA./MADERAS ARAUCO S.A
```

**Patrón con múltiples nomenclaturas:**
```
NOMENCLATURA : 1. [851]Ofíciese 2. [1391]Solicita Información Domicilio TGR JUZGADO : Juzgado de Letras de Illapel CAUSA ROL : C-153-2021 CARATULADO : BANCO DE ESTADO DE CHILE/SOCIEDAD COMERCIAL MOSA SPA
```

**Variantes observadas:**
- FOJA presente o ausente
- FOJA con texto: `FOJA: 24 .- veinticuatro .-` o sin: `FOJA: 31 .- .-`
- Tribunal con ordinal pegado: `17 Juzgado Civil de Santiago º` (el º se separa)
- Múltiples nomenclaturas: `1. [851]... 2. [1391]...`
- Nomenclatura con códigos numéricos: `[40]`, `[379]`, `[4764]`, `[83]`, `[851]`

**Acción:** EXTRAER metadata → REMOVER primera instancia (header del documento).
Si aparece repetido por página, remover todas las instancias posteriores.
**Metadata:**
```json
{
  "foja": 31,
  "foja_texto": "treinta y uno",
  "nomenclatura": [{"codigo": 40, "descripcion": "Sentencia"}],
  "juzgado": "17º Juzgado Civil de Santiago",
  "rol": "C-15200-2023",
  "caratulado": "MINERIA Y CONSTRUCCIONES CERRO ALTO LTDA./MADERAS ARAUCO S.A"
}
```

### 8.4 ARTEFACTO: ROL + Foja repetido por página

**Encontrado en:** fixture 01
**Frecuencia:** En sentencias/resoluciones largas, se repite en cada página.

**Patrón real (aparece después del marcador de página):**
```
-- 1 of 9 -- C-15200-2023 Foja: 1
```

La secuencia completa de page break en sentencias es:
```
[firma electrónica PJUD] -- N of M -- [ROL] Foja: [N]
```

**Acción:** Usar como indicador de page boundary → REMOVER el `ROL Foja: N` repetido.
El header NOMENCLATURA inicial ya capturó esta info.

### 8.5 ARTEFACTO: Encoding roto en pdf-parse (caracteres acentuados separados)

**Encontrado en:** fixtures 01, 06, 10 (todos pdf-parse)
**Severidad:** Alta en fixture 10, media en fixture 01, leve en fixture 06.

**HALLAZGO CRÍTICO:** Este problema ocurre en pdf-parse, NO solo en Document AI.
Contradice la asunción inicial del spec. El paso de normalización de encoding
DEBE aplicarse también a output de pdf-parse.

**Patrón:** Los caracteres acentuados (ó, á, é, í, ú, ñ) se extraen SEPARADOS
del word, dejando un espacio en el word y el carácter flotando:

```
Ejemplos reales de fixture 01 (pdf-parse):
  "presentaci n" → presentación (ó ausente del word)
  "representaci n" → representación
  "Miner a" → Minería (í ausente)
  "terminaci n" → terminación
  "gr as" → grúas (ú ausente)
  "se ala" → señala (ñ ausente)
  "a o" → año
  "d a" → día

Ejemplos reales de fixture 10 (pdf-parse, severo):
  "veintis is" → veintiséis
  "veintitr s" → veintitrés
  "habi ndose" → habiéndose
  "telem ticos" → telemáticos
  "do a" → doña
  "Roc o" → Rocío
  "c dula" → cédula
```

**Los caracteres acentuados flotantes aparecen en el texto pero desplazados:**
```
"en representaci n de Miner a y ó í Construcciones"
                                ↑ ↑ (ó e í flotando)
```

**Estrategia de corrección (2 pasos):**
1. Detectar y remover caracteres acentuados aislados (rodeados de espacios): ` ó `, ` á `, ` é `, ` í `, ` ú `, ` ñ `, ` Ó `, ` É `, ` Ñ `
2. Aplicar diccionario de correcciones conocidas del español jurídico:
   - `ci n` → `ción`, `si n` → `sión`, `ti n` → `tión`
   - `Miner a` → `Minería`, `a o` → `año`, `se ala` → `señala`
   - `d a` → `día`, `gr as` → `grúas`, `c dula` → `cédula`
   - Generalizar: vocal + espacio + consonante donde se espera tilde

**NOTA:** fixture 02 (demanda) tiene encoding diferente: `ı́` (i + combining acute)
y `Nici` en vez de `fici` (NotiNicación = Notificación). Esto es del procesador
de texto del abogado (LaTeX/Pages), no de pdf-parse.

### 8.12 TEXTOS QUE PARECEN ARTEFACTOS PERO SON CONTENIDO LEGAL (NO TOCAR)

Los siguientes patrones NO deben removerse:

1. **Formula "Proveyó Juez/a":**
   `Proveyó Juez (a) del Juzgado de Letras de Illapel que se individualiza en la firma electrónica avanzada estampada en la presente resolución.`
   → Contenido legal: indica quién dictó la resolución.

2. **Notificación por estado diario:**
   `En Illapel, a veintiuno de enero de dos mil veintiséis, se notificó por el estado diario, la resolución precedente. /rtd`
   → Contenido legal: fecha de notificación (computa plazos).

3. **Iniciales del funcionario:** `/rtd`, `/cvc`
   → Mínimo ruido, dejarlo.

4. **Header de escritos de abogados:**
   ```
   TRIBUNAL : 17° Juzgado Civil de Santiago
   ROL : C-15200-2023
   CARATULADO : "Minería y Construcciones..."
   CUADERNO : Principal
   _______________________________________________________________
   ```
   → Es parte del escrito del abogado, no un artefacto del sistema.
   Extractable como metadata pero NO remover.

5. **Header de receptor:**
   `TATIANA MUÑOZ MIMIZA RECEPTORA JUDICIAL 17º Juzgado Civil de Santiago`
   → Identifica al ministro de fe. Contenido legal relevante.

## 9. Hallazgos que modifican el diseño original

### 9.1 Encoding roto aplica a pdf-parse (no solo Document AI)

El refinamiento 2.2 del spec original asumía que el paso 5 (normalizar encoding)
solo aplicaba a Document AI. Los fixtures reales demuestran que **pdf-parse produce
encoding roto severo** con caracteres acentuados del español. El paso 5 DEBE
aplicarse a ambos métodos de extracción, con lógica diferenciada:

- **pdf-parse:** Caracteres acentuados separados del word (ej: `presentaci n` + `ó` flotando)
- **document-ai:** Artefactos OCR clásicos + marcas de agua + texto garbled de firmas

### 9.2 No se encontró "PODER JUDICIAL / REPÚBLICA DE CHILE" como header repetido

El Kanban original describía headers repetidos `PODER JUDICIAL + REPÚBLICA DE CHILE + 
nombre tribunal`. En los 15 fixtures reales **NO aparece este patrón**. Los headers
reales son:
- `NOMENCLATURA + JUZGADO + ROL + CARATULADO` (sistema PJUD)
- `TRIBUNAL + ROL + CARATULADO + CUADERNO` (escritos de abogados)

El normalizer NO debe buscar "PODER JUDICIAL" como header. Ajustar regex.

### 9.3 Las firmas electrónicas son simples (código + URL), no bloques largos

El Kanban describía bloques de 40-80 líneas con "Firmado electrónicamente por..." +
timbres + códigos Ley 20.886. En la realidad, la firma PJUD es una línea compacta:
`Código: XXXX Este documento tiene firma electrónica y su original puede ser validado en http://verificadoc.pjud.cl`

No hay nombre del firmante, cargo, ni fecha en la firma PJUD estándar. La info del
firmante aparece en el CONTENIDO del documento (ej: "DICTADA POR DOÑA ROCÍO PÉREZ
GAMBOA, JUEZA TITULAR"). El normalizer NO puede extraer firmante de la firma PJUD
— debe extraerlo del contenido si lo necesita (eso es tarea del section-detector 7.07c).

### 9.4 Secuencia real de page break en documentos PJUD

La secuencia completa de cambio de página en documentos generados por PJUD es:

```
[...contenido legal de la página...]
Código: XXDXXMLXXZH Este documento tiene firma electrónica y su original puede ser validado en http://verificadoc.pjud.cl -- N of M -- C-15200-2023 Foja: 1
[contenido legal de la siguiente página...]
```

Componentes del page break (en orden):
1. Firma electrónica PJUD (código + URL)
2. Marcador de página (`-- N of M --`)
3. ROL + Foja repetido (solo en sentencias/resoluciones)

El normalizer debe detectar esta secuencia completa para registrar page boundaries.

## 10. Notas de implementación

- `extracted_texts.full_text` es read-only. NUNCA se modifica en DB.
- El texto normalizado fluye al chunker (7.07b) en memoria.
- La metadata extraída de artefactos se almacena en `document_chunks.metadata`.
- Si el normalizer falla, se usa el texto original sin normalizar (fallback).
- Todas las funciones internas son puras (sin side effects, sin I/O).
